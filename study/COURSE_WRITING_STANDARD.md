# 课程编写规范

本规范用于后续所有课程生成与既有课程回改。目标是保证教材不是“代码片段集合”，而是适合 AI 初学者系统学习的完整课程材料。

## 1. 总原则

后续每一课都必须遵循：

```text
概念优先
术语讲透
流程清晰
重点标注
代码辅助
练习验收
```

课程内容的优先级：

```text
1. 专有名词和核心概念理解
2. 课程教授内容的详细说明
3. 概念之间的关系
4. 易错点和工程经验
5. 代码示例
6. 项目练习
```

【核心】代码和练习是帮助理解概念的工具，不是课程主体。

## 2. 每课固定结构

每一课主教材建议采用以下结构：

```text
# 第 N 课：课程标题

## 0. 本课阅读方式
说明本课应该怎么学，哪些内容先理解，哪些内容后实践。

## 0.1 概念讲义：主题术语与核心知识详解
集中解释本课所有关键术语和核心概念。

## 1. 本课目标
列出学习后能做什么、能解释什么。

## 2. 学习主线
用文字图或流程图串起本课内容。

## 3. 核心内容详解
按知识点展开讲解，先解释概念，再给必要示例。

## 4. 易错点与工程经验
标注常见误区、理解偏差、真实项目注意事项。

## 5. 代码示例
代码只服务于概念验证，不喧宾夺主。

## 6. 实践任务
练习用于巩固，不作为主要讲解方式。

## 7. 自测题
优先考概念理解，再考简单代码。

## 8. 阶段验收标准
明确学完后应能解释和完成什么。

## 9. 本课使用的信息源
列出官方文档、权威资料、参考来源。
```

实际章节可以根据内容调整标题编号，但必须保留“阅读方式”“概念讲义”“重点标注”“自我检查”这些学习支架。

## 3. 重点标注规范

教材中必须显式标注重点。

统一使用：

- `【核心】`：必须掌握的主干概念，后续课程会反复依赖。
- `【重点】`：重要知识点，考试、自测、项目中经常用到。
- `【易错】`：初学者最容易误解或写错的地方。
- `【工程经验】`：真实项目、生产环境、团队协作中需要注意的内容。
- `【类比】`：面向 Java 开发者的理解类比。
- `【检查】`：学习者需要自查的问题或清单。

示例：

```text
【核心】训练集用于学习参数，测试集用于最终评估泛化能力。

【易错】不要在全量数据上 fit 标准化器后再拆分训练集和测试集。
```

## 4. 术语解释规范

每个重要术语必须至少说明：

1. 中文名称。
2. 英文名称。
3. 一句话定义。
4. 在本课中的作用。
5. 和相近术语的区别。
6. 一个业务或 Java 开发类比。
7. 常见误解。

示例：

```text
特征 Feature

特征是模型用来做预测的输入信息。

在房价预测中，面积、房间数、房龄是特征。

【核心】特征必须在预测时真实可获得。

【易错】不能把标签本身或由标签计算出的字段作为特征。
```

## 5. 代码示例规范

代码示例必须遵循：

1. 先解释代码要验证什么概念。
2. 再给最小代码。
3. 再解释输入、输出和关键变量。
4. 不用代码替代概念讲解。
5. 不堆砌大量无解释代码。

代码示例优先采用：

```text
概念 -> 代码 -> 输出 -> 解释 -> 常见错误
```

## 6. 项目练习规范

项目练习必须包含：

1. 项目目标。
2. 练习对应的课程概念。
3. 验收标准。
4. 术语复述任务。
5. 易错点检查。
6. 可选拓展。

项目不是只要求“运行成功”，还要要求学习者能解释：

```text
为什么这么做？
输入是什么？
输出是什么？
这个步骤对应哪个概念？
如果错了，常见原因是什么？
```

## 7. 信息源规范

生成课程时应优先参考：

1. 官方文档。
2. 权威课程。
3. 主流框架文档。
4. 标准库文档。
5. 经典论文或官方教程。

每课必须在 `SOURCES.md` 中记录使用的信息源。

## 8. 进阶课程公式与理论注释规范

从进阶研究课程开始，凡涉及数学公式、理论结论、算法目标函数、优化更新式、概率模型、矩阵/张量分解、SDE/ODE、损失函数、复杂实验指标时，必须对公式和理论进行详细注释说明。

每个关键公式后必须尽量补齐以下内容：

```text
公式注释：
- 每个符号分别表示什么。
- 输入、输出、维度或张量形状是什么。
- 公式左边和右边分别在表达什么。
- 这个公式解决了什么问题。
- 它和模型训练、推理、泛化、稳定性或工程实现有什么关系。

理论注释：
- 该理论结论成立所需的前提假设。
- 直觉解释，而不是只给结论。
- 在深度学习或大模型中的具体对应场景。
- 常见误解和使用边界。
- 如果假设不成立，结论可能如何失效。
```

统一写法示例：

```text
$$
\Delta W = BA
$$

公式注释：

- $W$ 是原始权重矩阵。
- $\Delta W$ 是下游任务学习到的权重更新。
- $B \in \mathbb{R}^{d_{\text{out}}\times r}$ 负责把低维任务子空间映射回输出空间。
- $A \in \mathbb{R}^{r\times d_{\text{in}}}$ 负责把输入投影到低维任务子空间。
- 因为中间维度是 $r$，所以 $\operatorname{rank}(\Delta W)\le r$。

理论注释：

- 该公式假设下游任务所需更新集中在低维子空间。
- 如果任务需要大范围改变模型能力，过小的 $r$ 会造成结构性欠拟合。
- 它解释了 LoRA 为什么可以减少训练参数和优化器状态。
```

强制要求：

1. 不能只堆公式，必须解释公式中每个重要变量。
2. 不能只说“显然”“可得”“容易证明”，必须说明关键推导逻辑。
3. 不能只给抽象理论，必须说明它对应的大模型训练、微调、推理或评估场景。
4. 对 2024-2026 年前沿方法，必须说明其改动的是参数化、优化器、初始化、数据、目标函数、推理系统还是评估协议。
5. 代码实验中的数学指标也必须解释，例如 loss、rank、trace、top eigenvalue、KL、ELBO、reward、throughput、latency。

进阶课程的固定结构在原有课程结构基础上追加：

```text
## 公式与理论注释
穿插在每个公式或理论段落之后，不单独堆到最后。

## 前沿研究进展
至少包含 2 项 2024-2026 年研究进展，并说明数学或工程改动点。

## MCP 调用点设计
明确调用目的、建议工具、建议查询、预期学习收获。

## 课后研究课题
包含开放问题、论文查阅要求、实现或分析任务、验收输出。
```

【核心】后续进阶课程必须做到“公式可读、理论可解释、实验可复现、前沿可追踪”。公式不是装饰，必须成为学员理解模型机制和研究论文的工具。

## 9. 后续课程执行要求

从第 4 课开始，所有新课程都必须按本规范生成。

已经生成的第 1 课和第 2 课需要回改：

- 增加“本课阅读方式”。
- 增加“概念讲义”。
- 补齐术语解释。
- 增加重点标注。
- 强化内容说明。
- 保留代码与练习，但放在概念之后。
