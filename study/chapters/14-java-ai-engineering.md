# 第 14 课：Java AI 工程化：流式输出、权限、日志、成本与评估

## 0. 本课阅读方式

本课是 AI 应用从 Demo 走向生产系统的关键。你需要把大模型调用当成一个高成本、可能失败、需要审计、需要权限控制的外部依赖，而不是一个普通工具类。

学习主线：

```text
流式体验 -> 权限和安全 -> 日志和审计 -> 成本和限流 -> 缓存和降级 -> 质量评估 -> 监控和运维
```

【核心】AI 工程化不是“能调用模型”，而是“可控、可观测、可评估、可回滚、可治理”。

## 0.1 概念讲义：Java AI 工程化术语详解

### 0.1.1 AI 工程化

AI 工程化是把模型能力稳定接入真实业务系统的工程实践。

它关注：用户体验、权限边界、成本预算、错误兜底、日志审计、质量评估、安全风险、持续监控。

【核心】AI 应用上线后的问题，往往不是“模型不会答”，而是“系统不可控”。

### 0.1.2 Streaming 流式输出

流式输出是模型边生成边返回。它能提升首字响应和长回答体验，但会带来连接中断、前端取消、半截答案、日志聚合、敏感内容中途输出等问题。

【重点】流式输出提升体验，但也增加状态管理和异常处理复杂度。

### 0.1.3 Authorization 权限控制

AI 系统权限包括：谁可以使用 AI、访问哪些知识库、调用哪些工具、执行哪些写操作、查看哪些日志。

【核心】权限必须在应用层和检索层执行，不能只靠 Prompt 约束模型。

### 0.1.4 Audit Log 审计日志

审计日志记录 AI 调用过程。应记录用户、会话、Prompt 版本、模型、检索文档、工具调用、输入输出 token、延迟、错误、成本。

【工程经验】没有日志，就无法排查错误、核算成本、定位责任。

### 0.1.5 Cost Control 成本控制

AI 调用成本通常和 token、模型、调用次数、工具调用、检索等相关。

成本控制手段：限制输入长度、限制输出长度、缓存、模型分级、用户配额、项目预算、批处理、降级策略。

### 0.1.6 Rate Limit 限流

限流用于保护模型供应商、系统资源和预算。常见维度包括每用户每分钟、每租户每天、每接口并发、每模型调用量、token 预算。

【重点】AI 功能必须有限流，否则成本和延迟很容易失控。

### 0.1.7 Retry and Backoff

模型 API 可能因为限流、服务错误、网络超时、输出格式错误而失败。退避重试应设置最大次数，并避免重复执行写操作工具。

### 0.1.8 Caching 缓存

缓存适合相同问题、稳定知识库答案、Embedding 结果、文档解析结果、工具只读查询。

【易错】AI 答案缓存必须考虑用户权限和知识版本。

### 0.1.9 Evaluation 评估

AI 评估包括 Prompt 评估、RAG 检索评估、回答正确性评估、工具调用评估、安全评估、用户反馈评估。

【核心】没有评估集，AI 应用无法稳定迭代。

### 0.1.10 Observability 可观测性

可观测性包括日志、指标、Trace、成本、延迟、错误率、token 使用、工具调用链路。

【重点】AI 调用链更长，更需要全链路追踪。

### 0.1.11 Guardrails 护栏

护栏包括输入过滤、输出校验、敏感信息检测、工具权限、人工确认、拒答策略。

### 0.1.12 本课重点标注汇总

【核心】AI 工程化目标是可控、可观测、可评估、可回滚。

【核心】权限控制不能依赖模型自觉。

【核心】日志和评估是持续优化的基础。

【重点】流式输出提升体验，但要处理取消、中断和日志聚合。

【重点】成本控制必须从 token、模型、缓存、限流多层设计。

【易错】不要把模型输出直接写入业务系统。

## 1. Java AI 生产架构

```text
Controller
  -> Auth / RateLimit
  -> AI Application Service
      -> Prompt Builder
      -> Retriever / Tool Executor
      -> Model Client
      -> Output Validator
  -> Audit Log / Metrics / Trace
  -> Response
```

## 2. 流式输出设计

后端需要处理 SSE 或 WebSocket、用户取消、超时、错误事件、完整答案落库、中途敏感内容处理。

【工程经验】流式返回给用户的同时，要在服务端聚合完整响应用于日志和审计。

## 3. 权限设计

| 维度 | 说明 |
|---|---|
| 功能权限 | 是否可使用 AI |
| 数据权限 | 可检索哪些文档 |
| 工具权限 | 可调用哪些工具 |
| 操作权限 | 是否可执行写操作 |
| 日志权限 | 是否可查看历史对话 |

## 4. 成本治理

建议记录：

```text
input_tokens
output_tokens
model
request_count
tool_call_count
retrieved_chunk_count
estimated_cost
```

策略：免费用户用低成本模型，高风险任务用高能力模型，先检索再调用，超长输入先摘要，重复问题走缓存。

## 5. 评估体系

至少建立 RAG 问题集、Prompt 回归测试集、工具调用测试集、拒答测试集、安全测试集。

【核心】每次改 Prompt、模型、chunk 策略、top-k，都应该回归评估。

## 6. 常见误区

1. Demo 能跑就上线。
2. 没有限流和预算。
3. 没有 Prompt 版本。
4. 没有检索日志。
5. 没有用户权限过滤。
6. 不记录工具调用。
7. 不做评估集。
8. 不设置降级方案。
