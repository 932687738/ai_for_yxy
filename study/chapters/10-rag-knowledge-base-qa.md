# 第 10 课：RAG 知识库问答系统

## 0. 本课阅读方式

本课把第 8 课的 Prompt、第 9 课的 Embedding 和向量数据库串成一个完整系统：RAG 知识库问答。

学习本课时，不要只关注“怎么调 API”，而要理解：

```text
为什么需要 RAG？
RAG 和微调有什么区别？
文档如何入库？
用户问题如何检索？
检索结果如何进入 Prompt？
模型如何基于资料回答？
如何评估 RAG 质量？
如何处理权限、更新、引用、幻觉和兜底？
```

【核心】RAG 的本质是“先检索，再生成”：把外部知识检索出来，作为上下文交给大模型回答。

## 0.1 概念讲义：RAG 知识库问答术语详解

### 0.1.1 第 10 课到底在学什么

普通大模型只依赖自身参数和输入上下文。

企业知识库问答需要回答：

```text
基于企业制度、产品手册、项目文档、FAQ 等私有资料的问题。
```

这些资料可能：

- 不在模型训练数据中。
- 经常更新。
- 有权限限制。
- 需要引用来源。
- 不能让模型胡编。

RAG 用检索解决这些问题：

```text
用户问题
  -> 检索相关资料
  -> 将资料放进 Prompt
  -> 大模型基于资料回答
```

【重点】RAG 不是让模型“记住”知识，而是让模型“临时读取”相关知识。

### 0.1.2 RAG

RAG 是 Retrieval-Augmented Generation，检索增强生成。

它由两部分组成：

```text
Retrieval：检索相关资料。
Generation：基于资料生成回答。
```

【核心】RAG 用外部知识增强大模型回答能力。

### 0.1.3 Knowledge Base

知识库是被检索的数据集合。

可以包括：

- PDF。
- Word。
- Markdown。
- HTML。
- FAQ。
- 数据库记录。
- 接口返回数据。
- 代码文档。

【重点】知识库质量直接决定 RAG 上限。

### 0.1.4 Document Loader

Document Loader 是文档加载器。

作用：

```text
从文件、网页、数据库、对象存储中读取原始内容。
```

【工程经验】文档加载不只是读取文本，还要保留来源、标题、页码、权限等 metadata。

### 0.1.5 Text Splitter

Text Splitter 是文本切分器。

作用：

```text
把长文档切成适合检索和上下文拼装的 chunk。
```

【核心】切分质量直接影响检索质量。

### 0.1.6 Chunk

Chunk 是文档片段。

好的 chunk 应该：

- 语义相对完整。
- 长度适中。
- 能独立回答某类问题。
- 保留来源信息。
- 不跨越太多无关主题。

【易错】chunk 不是越短越好，也不是越长越好。

### 0.1.7 Embedding

Embedding 把 chunk 和用户问题转成向量。

RAG 中通常有两类向量：

```text
文档 chunk embedding
查询 query embedding
```

【核心】查询和文档必须在同一语义向量空间里比较相似度。

### 0.1.8 Vector Store

Vector Store 是向量存储。

作用：

- 存储 chunk embedding。
- 存储 chunk 文本。
- 存储 metadata。
- 执行相似度搜索。
- 支持过滤和索引。

### 0.1.9 Retriever

Retriever 是检索器。

它接收用户问题，返回相关文档片段。

检索器可能使用：

- 向量检索。
- 关键词检索。
- 混合检索。
- 元数据过滤。
- Rerank。

【核心】Retriever 决定大模型能看到哪些资料。

### 0.1.10 Context

Context 是送给模型的上下文资料。

RAG 中通常包括：

- 用户问题。
- 检索到的 chunk。
- 来源信息。
- 回答规则。
- 输出格式。

【重点】上下文不是越多越好，相关、准确、简洁更重要。

### 0.1.11 Grounded Answer

Grounded Answer 是基于资料的回答。

它要求：

```text
回答必须能从提供资料中找到依据。
```

【核心】企业知识库问答应该优先追求 grounded answer，而不是自由发挥。

### 0.1.12 Citation

Citation 是引用来源。

例如：

```text
依据：《员工休假制度》第 3.2 节
```

引用的价值：

- 提高可信度。
- 方便用户核验。
- 便于审计。
- 降低幻觉风险。

### 0.1.13 Query Rewrite

Query Rewrite 是查询改写。

用户问题可能不适合直接检索。

例如：

```text
“这个怎么申请？”
```

需要结合上下文改写为：

```text
“年假申请流程是什么？”
```

【重点】多轮对话 RAG 中，查询改写很常见。

### 0.1.14 Multi-query Retrieval

Multi-query Retrieval 是把一个问题改写成多个查询，从不同角度检索。

适合：

- 用户问题模糊。
- 文档表达多样。
- 需要提高召回。

代价：

- 成本增加。
- 延迟增加。
- 可能引入噪声。

### 0.1.15 Rerank

Rerank 是对初步检索结果重新排序。

常见流程：

```text
向量检索 top 20
  -> rerank
  -> 取 top 5
  -> 放入 Prompt
```

【重点】Rerank 常用于提高上下文质量。

### 0.1.16 Answer Generation

Answer Generation 是生成回答阶段。

模型输入：

- 用户问题。
- 检索资料。
- 回答规则。
- 输出格式。

模型输出：

- 结论。
- 引用依据。
- 不确定点。

### 0.1.17 Refusal / No Answer

当资料不足时，模型应该拒答或说明无法确认。

例如：

```text
根据提供资料无法确认该问题。
```

【核心】RAG 系统必须允许“不知道”，否则容易幻觉。

### 0.1.18 Permission Filtering

权限过滤是根据用户权限过滤可检索资料。

例如：

```text
普通员工只能检索公开制度。
HR 可检索 HR 内部制度。
项目成员只能检索自己项目文档。
```

【工程经验】权限过滤应该发生在检索阶段，而不是只靠模型遵守指令。

### 0.1.19 Freshness

Freshness 是知识新鲜度。

知识库文档会更新，RAG 系统需要处理：

- 新增文档。
- 修改文档。
- 删除文档。
- 旧向量失效。
- 版本切换。

【重点】文档更新后需要重新切分、重新 embedding、重新入库或更新索引。

### 0.1.20 RAG Evaluation

RAG 评估包括两层：

```text
检索评估：有没有找对资料？
生成评估：有没有基于资料正确回答？
```

常见问题：

- 检索没找到正确资料。
- 找到了但排名太低。
- 找到了但模型没用。
- 模型引用错。
- 模型超出资料编造。

【核心】RAG 失败时要先定位是检索失败还是生成失败。

### 0.1.21 本课重点标注汇总

【核心】RAG = 检索 + 生成。

【核心】RAG 用外部知识增强模型，不是把知识训练进模型参数。

【核心】文档切分、Embedding、检索、Prompt 组装共同决定 RAG 质量。

【核心】权限过滤必须在检索阶段执行。

【核心】资料不足时必须允许拒答。

【重点】引用来源能提高可信度和可审计性。

【重点】RAG 评估要分开看检索质量和生成质量。

【易错】不要把所有检索结果无脑塞进 Prompt。

【易错】不要只优化 Prompt，而忽略文档质量和检索质量。

## 2. RAG 总体架构

```text
离线入库流程：

文档源
  -> 文档加载
  -> 文本清洗
  -> 文档切分
  -> 生成 Embedding
  -> 写入向量数据库

在线问答流程：

用户问题
  -> 权限识别
  -> 查询改写
  -> 生成 Query Embedding
  -> 检索相关 Chunk
  -> Rerank / 过滤
  -> Prompt 组装
  -> 大模型生成回答
  -> 输出解析和引用
```

【核心】RAG 是离线入库和在线检索生成两条链路共同组成的系统。

## 3. RAG 和微调的区别

| 对比 | RAG | 微调 |
|---|---|---|
| 目标 | 接入外部知识 | 改变模型行为或能力 |
| 数据更新 | 文档更新后重新入库 | 需要重新训练或继续训练 |
| 适合 | 知识库问答、制度问答 | 固定格式、风格、垂直任务 |
| 可溯源 | 容易引用文档 | 不天然引用来源 |
| 成本 | 通常较低 | 通常较高 |

【重点】企业知识库问答通常优先 RAG，不是优先微调。

## 4. 离线入库流程

### 4.1 文档加载

文档来源：

- PDF。
- Word。
- Markdown。
- HTML。
- 数据库。
- 对象存储。
- 企业知识平台。

加载时要保留：

- 文档 ID。
- 标题。
- 来源。
- 页码。
- 章节。
- 权限。
- 更新时间。

### 4.2 文本清洗

清洗内容：

- 页眉页脚。
- 乱码。
- 重复目录。
- 无意义空白。
- OCR 错误。
- 水印。

【工程经验】脏文档会直接污染检索结果。

### 4.3 文档切分

切分策略要考虑文档结构。

```text
优先按标题、段落、条款切分。
必要时使用 overlap。
避免跨主题混切。
```

### 4.4 向量化与入库

每个 chunk 生成 embedding 后写入向量库。

记录结构：

```text
chunk_id
doc_id
content
embedding
metadata
```

## 5. 在线问答流程

### 5.1 用户问题处理

用户问题可能：

- 太短。
- 指代不清。
- 包含错别字。
- 有多轮上下文。
- 包含恶意指令。

必要时需要：

- 查询改写。
- 意图识别。
- 权限判断。
- 安全过滤。

### 5.2 检索

检索步骤：

```text
用户问题 -> query embedding -> vector search -> top-k chunks
```

可以加入：

- metadata filter。
- hybrid search。
- rerank。
- score threshold。

### 5.3 Prompt 组装

Prompt 应包含：

```text
角色
回答规则
用户问题
检索资料
引用要求
拒答规则
输出格式
```

示例规则：

```text
请只基于资料回答。
如果资料中没有答案，请回答“根据提供资料无法确认”。
每个结论必须给出引用来源。
```

### 5.4 生成与后处理

后处理包括：

- JSON 校验。
- 引用校验。
- 敏感信息过滤。
- 答案长度控制。
- 日志记录。
- 失败兜底。

## 6. RAG 质量问题定位

RAG 答错时，先问：

```text
正确资料是否被检索出来？
```

如果没有：

- 文档是否入库？
- chunk 是否切坏？
- embedding 是否合适？
- top-k 是否太小？
- query 是否需要改写？
- 权限过滤是否误伤？

如果检索到了：

- Prompt 是否清晰？
- 上下文是否太长？
- 模型是否忽略资料？
- 引用是否错误？
- 输出格式是否不稳定？

【核心】RAG 调优要分层定位。

## 7. Java/Spring 落地架构

```text
Spring Boot API
  |
  +-- 用户认证与权限
  +-- 文档管理
  +-- 文档解析与切分
  +-- Embedding 调用
  +-- 向量数据库写入
  +-- 检索服务
  +-- Prompt 组装
  +-- LLM 调用
  +-- 日志审计
  +-- 成本统计
```

常用组件：

- Spring Boot。
- Spring AI。
- PostgreSQL + pgvector。
- Redis。
- 对象存储。
- 大模型 API。

【工程经验】RAG 不是一个接口，而是一套文档、检索、生成、权限和评估系统。

## 8. RAG 评估设计

### 8.1 检索评估

指标：

- Recall@k。
- Precision@k。
- MRR。
- 命中文档率。
- 命中 chunk 率。

### 8.2 生成评估

指标：

- 答案正确性。
- 是否基于资料。
- 引用是否正确。
- 是否拒答合理。
- 格式是否稳定。

### 8.3 测试集

需要构造：

```text
问题
标准答案
应命中文档
应命中 chunk
权限角色
是否应拒答
```

## 9. 常见误区

1. 把 RAG 当成“向量库 + 大模型”的简单拼接。
2. 不做文档清洗。
3. chunk 策略随意。
4. 不存 metadata。
5. 不做权限过滤。
6. 只看最终回答，不看检索命中。
7. 不允许模型拒答。
8. 不处理文档更新。
9. 不记录问题、检索结果和回答日志。
