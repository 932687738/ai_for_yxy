## 0.1 概念讲义：Embedding、语义搜索与向量数据库术语详解

### 0.1.1 第 9 课到底在学什么

第 8 课讲 Prompt，本课讲模型如何“找到相关资料”。

知识库问答通常不是把所有文档直接塞给大模型，因为：

- 文档太长。
- 上下文窗口有限。
- 成本高。
- 噪声多。
- 模型容易忽略关键信息。

更常见做法是：

```text
先检索相关片段
再把相关片段放进 Prompt
让模型基于片段回答
```

这个检索过程的核心就是 Embedding 和向量搜索。

【核心】Embedding 是 RAG 的检索基础。

### 核心讲解（课程模式 · 0.1.1 · 2026-05-15）

**链路位置**：第 8 课你把「问法」固化为 **Prompt**；本课解决 **「检索段从哪来」**。在多数 RAG 里，**检索失败**（召回错段、漏段）会直接表现为 **答非所问**，再好的生成都救不回来。

**三条约束**（先记骨架）：**(1)** **问题与文档片段** 要进 **同一向量空间**（通常 = **同一 Embedding 模型**）；**(2)** 放进 Prompt 的只是 **Top-k 片段**，不是整库；**(3)** **Chunk / 元数据** 设计本课后面单讲，它们决定 **可检性** 与 **权限过滤**。

**跟练（约 10 分钟）**：在白板或笔记上画三框：**用户问 → 向量 → 近邻片段 → 拼进 Prompt → LLM**，框下各写 **一个失败例子**（例如 **模型不一致**、**k 太小漏召回**）。

**自测**：为什么「全库原文塞进一次请求」在工程上通常不行？—— **窗口与成本** 不够，且 **噪声** 大、模型易 **走神**。

### 0.1.2 Embedding

Embedding 是把文本、图片、商品、用户等对象转换成向量。

文本例子：

```text
"如何申请年假？"
  -> [0.12, -0.08, 0.33, ...]
```

向量中的每个数字没有简单的人工含义，但整体向量能表示语义位置。

【核心】Embedding 把不可直接计算的语义，变成可以计算相似度的数字向量。

### 核心讲解（课程模式 · 0.1.2 · 2026-05-15）

**工程一句话**：Embedding 是 **「可检索坐标」**——文本进入模型后变成 **固定维向量**，之后在 **向量空间** 里做 **近邻搜索**。单个维度 **一般没有可读出的人话含义**，**别当特征工程那样逐列解释**。

**与业务的关系**：同一个 **Embedding 模型** 下，**「意思近」** 往往表现为 **向量近**（具体用 **余弦 / 点积 / 欧氏** 由模型与库约定）；**换模型 = 换几何**，旧索引通常 **要重建或迁移方案**。

**跟练**：各写一句 **同义**、一句 **反义** 的业务问法（如工单场景），思考：若只做 **关键词**，哪条会 **漏召**？—— 通常 **同义改写** 依赖 **语义检索** 才稳。

**自测**：Embedding 与「手工 TF-IDF 向量」最大的心智差别？—— **表示来自深度模型预训练**，对 **改写、跨说法** 更鲁棒（代价是 **依赖模型与算力**）。

### 0.1.3 Embedding Model

Embedding Model 是专门把输入转换成向量的模型。

它和聊天模型不同：

```text
聊天模型：输入文本，输出文本。
Embedding 模型：输入文本，输出向量。
```

【重点】做语义搜索时，文档和查询通常必须使用同一个 Embedding 模型生成向量。

### 核心讲解（课程模式 · 0.1.3 · 2026-05-15）

**排雷**：**「用 A 模型 embed 文档，用 B 模型 embed 问题」** 在多数产品里等于 **自选 bug**：两路向量 **不在同一表示空间**，**近邻分数没有可比性**。**升级 Embedding** 时常见动作：**全量重扫（re-embed）+ 换索引**。

**选型粗原则**：**(1)** 领域是否 match（通用 vs 代码 vs 双语）；**(2)** **延迟与 QPS**；**(3)** **向量维度** 与 **库是否支持**；**(4)** 官方推荐的 **相似度**（有的训练目标偏向 **余弦归一**）。

**跟练**：打开你们或云厂商文档，抄三行到笔记：**模型 id**、**输出维度**、**推荐 metric**。

**自测**：聊天模型能否直接当 Embedding 用？—— **有时可以（取某层向量）但要有协议**；生产更常见的是 **专用 embedding API**，**别混用未文档约定的隐藏状态**。

### 0.1.4 Vector 向量

向量是一组数字。

在 Embedding 场景中：

```text
向量代表语义位置。
```

语义接近的文本，向量方向或距离通常更接近。

例如：

```text
"年假怎么申请"
"如何提交休假流程"
```

字面不同，但语义接近。

### 核心讲解（课程模式 · 0.1.4 · 2026-05-15）

**直觉**：把向量想成 **高维空间里的一支箭头**。**方向** 近似 = **主题、意图** 近似；**长度** 有时与 **文本长短 / 训练归一化** 有关——因此有的系统 **先 L2 归一** 再用 **点积等价余弦**。

**反例心智**：**「向量近」≠「业务上该答这份文档」**——还可能 **近邻是噪声**、**是没权限的文档**、或 **chunk 切断** 导致 **语义残片**。后面 **Rerank、权限过滤、引用校验** 仍是必修。

**跟练**：对教材里两句 **年假** 改写，再额外写一句 **表面相似但主题不同** 的干扰句（例如 **「年假薪酬怎么报税」** 若政策库只有 **流程**），想想：**仅靠向量** 会不会 **误召**？—— 体会 **混合检索 + 元数据过滤** 的价值。

**自测**：为什么二维、三维画图直觉 **会在高维失灵**？—— **距离集中、邻域结构更复杂**；工程上以 **离线评测（Recall@k）** 为准，不靠 **肉眼**。

### 0.1.5 Semantic Search 语义搜索

语义搜索根据意思查找，而不是只匹配关键词。

关键词搜索：

```text
查询“年假”
只容易找到包含“年假”的文档。
```

语义搜索：

```text
查询“怎么请假”
也可能找到“年假申请流程”。
```

【核心】语义搜索解决“表达不同但意思接近”的检索问题。

### 核心讲解（课程模式 · 0.1.5 · 2026-05-15）

**边界**：语义搜索擅长 **释义、口语化、跨说法**；对 **精确编号匹配**（订单号、文件名、错误码全文）未必赢 **关键词 / BM25**。产品形态上往往是 **「语义主召回 + 关键词补洞」** 或 **`### 0.1.21` 混合检索**。

**失败模式**：**(1)** **过召**：top-k 里 **大体相关但不可用**；**(2)** **漏召**：措辞略偏 **就出榜**；**(3)** **域外**：问题 **不在库里** 仍 **硬近邻**——要结合 **拒答、置信阈值、Rerank**。

**跟练**：把用户口语「**老板不让休**」映射到制度里可能的 **规范表述**（若制度写 **「审批未通过」**），理解 **关键词漏、语义补**。

**自测**：语义搜索与「全文 LIKE」的本质差别？—— **匹配对象** 从 **字符子串** 换成 **向量邻域**。

### 拓展阅读

- **`### 0.1.6`**：关键词 **何时仍是 king**。  
- **Elasticsearch / OpenSearch**：**dense_vector + BM25** 的混合路径常见于企业。

### 联想扩展

- **与多语言**：跨语言检索要看 **模型是否多语对齐**；**勿默认「自动翻译就够」**。

### 0.1.6 Keyword Search 关键词搜索

关键词搜索依赖词项匹配。

优点：

- 精确。
- 可解释。
- 对专业术语、编号、代码、SKU 很有效。

缺点：

- 同义表达难召回。
- 依赖分词。
- 不理解语义。

【工程经验】真实系统常使用混合检索：关键词搜索 + 向量搜索。

### 核心讲解（课程模式 · 0.1.6 · 2026-05-15）

**关键词的长板**：**精确、可解释、对符号敏感**（SKU、接口名、`error_code=0x5a`）。**短板**：**分词与词形**、**同义扩展** 要人工维护，**成本高**。

**落地建议**：**(1)** **元数据字段**（部门、版本、租户）优先 **过滤** 再向量；**(2)** **强标识** 走 **倒排**；**(3)** **自然语言意图** 走 **向量**；**(4)** 结果侧用 **Rerank 统一打分**（见 **`### 0.1.22`**）。

**跟练**：列你们库里的 **三类字段**：**必须精确匹配**、**可范围过滤**、**适合向量**——各举一例。

**自测**：只做向量、不做关键词，哪种查询最先翻车？—— **「含 `NullPointerException` 的堆栈片段」** 这类 **低频长串符号** 往往要 **子串/ ngram 支持**。

### 0.1.7 Similarity 相似度

相似度衡量两个向量有多接近。

常见方式：

- Cosine similarity
- Dot product
- Euclidean distance

【重点】不同向量数据库或模型可能推荐不同距离度量，要保持一致。

### 核心讲解（课程模式 · 0.1.7 · 2026-05-15）

**三件事绑定**：**Embedding 训练目标**、**索引算法假设**、**线上 metric**。**私自把余弦换成欧氏** 却不 **重算索引**，等于 **偷偷改了排序**。

**粗略对照**：**余弦** 看 **方向**（常用于 **归一化** 向量）；**点积** 若向量 **已单位化** 与余弦 **单调相关**；**欧氏** 对 **模长** 敏感。**ANN（`### 0.1.17`）** 是在 **牺牲少量精确性** 换 **延迟**。

**跟练（约 10 分钟）**：

```bash
python ai-learning/09-embedding-semantic-search-vector-db/scripts/lesson09_cosine_dot_stub_demo.py
```

看 **「同向放大」**：**点积变大了**，**余弦不变**——体会 **模长对点积的影响**。

**自测**：为什么 **索引版本** 要和 **metric** 一起发版？—— **近邻图 / 分区** 依赖 **距离定义**，不一致则 **召回不可比**。

- **FAISS / pgvector 文档**：每种 **opclass / space** 对应一类 **距离**。

### 0.1.8 Cosine Similarity

余弦相似度衡量两个向量方向是否接近。

在语义检索中常用。

【核心】语义相近通常意味着向量方向相近。

### 核心讲解（课程模式 · 0.1.8 · 2026-05-15）

**公式心智**（不必背）：**cos(θ) = (a·b) / (||a||·||b||)**。**范围** 一般在 **[-1, 1]**（数值噪声可能略越界）。**1** 表 **同向**，**0** 表 **正交**，**-1** 表 **反向**。

**何时爱用余弦**：向量 **模长不稳定**（短文 vs 长文 embed）、或模型在训练里 **以方向为主**。**注意**：**余弦 + 未归一化索引** 若实现细节混了 **内积近似**，要按厂商说明 **核对**。

**跟练**：继续用上节脚本，看 **「正交」「反向」** 两组：**点积与余弦** 各是多少？能否 **口头解释** 为什么 **正交为 0**？—— **无投影分量**。

**自测**：余弦相同、**欧氏距离** 不同的情况？—— **同一方向不同长度** 的向量。

### 拓展阅读

- **`### 0.1.9`**：点积与 **「已归一化」** 前提。  
- **向量库**：有的 API 返回 **distance** 而非 **similarity**，要弄清 **单调方向**（越大越好还是越小越好）。

### 联想扩展

- **与安全**：**极大向量范数** 有时是 **对抗样本信号**；生产要有 **输入长度与异常检测**（此处仅点题）。

### 0.1.10 Vector Database 向量数据库

向量数据库用于存储和检索向量。

它不仅存向量，还通常存：

- 文档 ID。
- 文档片段文本。
- 元数据。
- 来源文件。
- 权限字段。
- 时间信息。

【核心】向量数据库解决“大量向量如何高效相似度检索”的问题。

### 0.1.11 Collection / Table

向量数据库通常把向量组织到 collection 或 table 中。

类比关系：

```text
关系数据库：database / table / row
向量数据库：collection / point / vector
PostgreSQL + pgvector：table / row / vector column
```

### 0.1.12 Metadata / Payload

Metadata 是和向量一起存储的业务信息。

例如：

```json
{
  "doc_id": "policy-001",
  "title": "年假管理制度",
  "department": "HR",
  "permission": "employee",
  "chunk_index": 3
}
```

【核心】元数据用于过滤、权限控制、结果展示和溯源。

### 0.1.13 Chunk

Chunk 是文档切分后的片段。

为什么要切分？

- 文档太长。
- Embedding 模型有输入长度限制。
- 检索需要更精确片段。
- 大模型上下文窗口有限。

【重点】Chunk 切太大，噪声多；切太小，上下文不足。

### 0.1.14 Chunking Strategy

切分策略决定如何把文档拆成片段。

常见策略：

- 按固定字数。
- 按标题层级。
- 按段落。
- 按语义边界。
- 滑动窗口重叠。

【工程经验】文档结构越强，越应该利用标题、段落、章节切分，而不是简单按字数硬切。

### 0.1.15 Overlap

Overlap 是相邻 chunk 之间的重叠内容。

作用：

- 防止关键信息被切断。
- 保留上下文连续性。

代价：

- 增加存储。
- 增加检索重复。
- 增加成本。

### 0.1.16 Top-k

Top-k 表示返回最相似的前 k 条结果。

例如：

```text
top_k = 5
```

表示返回最相关的 5 个文档片段。

【重点】k 太小可能漏召回，k 太大可能引入噪声。

### 0.1.17 ANN

ANN 是 Approximate Nearest Neighbor，近似最近邻搜索。

因为精确搜索大量向量成本高，向量数据库常用近似算法提升速度。

【工程经验】向量搜索通常在准确率、速度、索引大小之间做权衡。

### 0.1.18 HNSW

HNSW 是常见 ANN 索引算法。

它通过图结构加速相似向量搜索。

pgvector、Qdrant、Milvus 等都支持或使用类似 ANN 索引能力。

### 0.1.19 IVFFlat

IVFFlat 是另一类向量索引思路，通过聚类分区减少搜索范围。

在 pgvector 中，HNSW 和 IVFFlat 都是常见索引选择。

### 0.1.20 pgvector

pgvector 是 PostgreSQL 的向量扩展。

它让 PostgreSQL 能存储向量并执行相似度搜索。

适合：

- 已经使用 PostgreSQL。
- 数据规模中小。
- 希望关系数据和向量数据放在一起。
- 需要 SQL、事务、权限集成。

【核心】pgvector 是 Java/Spring 系统落地 RAG 的常见选择。

### 0.1.21 Hybrid Search 混合检索

混合检索结合关键词搜索和向量搜索。

适合：

- 既要语义召回。
- 又要精确匹配编号、术语、代码、产品型号。

【工程经验】企业知识库中，纯向量检索经常不如混合检索稳定。

### 0.1.22 Rerank 重排序

Rerank 是对初步检索结果再次排序。

流程：

```text
向量搜索召回 top 20
  -> reranker 精排
  -> 选 top 5 给大模型
```

【重点】Rerank 可以提升最终送入大模型的上下文质量。

### 0.1.23 Recall@k

Recall@k 衡量正确结果是否出现在前 k 个检索结果中。

例如：

```text
用户问题的正确文档出现在 top 5 中
```

说明 Recall@5 命中。

### 0.1.24 本课重点标注汇总

【核心】Embedding 把语义转换成向量。

【核心】语义搜索通过向量相似度找相关内容。

【核心】向量数据库解决大规模向量存储和相似度检索。

【核心】Chunking 直接影响 RAG 检索质量。

【核心】Metadata 是权限、过滤、溯源和展示的基础。

【重点】关键词搜索和语义搜索各有优缺点，真实系统常用混合检索。

【重点】Top-k、相似度阈值、重排序都会影响最终回答质量。

【易错】向量相似不等于答案一定正确，还需要上下文验证。

## 2. 语义搜索流程

```text
文档入库：
  原始文档
    -> 清洗
    -> 切分 chunk
    -> 生成 embedding
    -> 存入向量数据库

用户查询：
  用户问题
    -> 生成 query embedding
    -> 向量相似度搜索
    -> 取回 top-k chunk
    -> 可选 rerank
    -> 送给大模型回答
```

【核心】入库和查询必须使用兼容的 Embedding 表示。

## 3. 关键词搜索 vs 语义搜索

| 对比 | 关键词搜索 | 语义搜索 |
|---|---|---|
| 匹配方式 | 词项匹配 | 向量相似度 |
| 优点 | 精确、可解释 | 能理解同义表达 |
| 缺点 | 同义词召回差 | 可能召回语义相关但不精确内容 |
| 适合 | 编号、代码、术语 | 自然语言问题 |

【工程经验】企业系统中优先考虑混合检索，而不是非此即彼。

## 4. Chunk 设计

切分文档时要考虑：

1. 文档结构。
2. 段落边界。
3. 标题层级。
4. 表格。
5. 列表。
6. 语义完整性。
7. 模型上下文窗口。

示例：

```text
制度标题
  -> 一级章节
  -> 二级章节
  -> 段落
```

【重点】好的 chunk 应该能独立回答一个小问题，同时保留必要上下文。

## 5. 向量数据库设计

一条向量记录通常包含：

```text
id
embedding
content
metadata
created_at
updated_at
```

metadata 示例：

```json
{
  "doc_id": "hr-policy-001",
  "title": "员工休假管理制度",
  "source": "hr_policy.pdf",
  "department": "HR",
  "permission": "employee",
  "chunk_index": 5
}
```

【核心】没有 metadata，检索结果难以权限过滤和结果溯源。

## 6. pgvector 概念示例

pgvector 让 PostgreSQL 支持向量字段。

概念表结构：

```sql
CREATE TABLE document_chunks (
    id bigserial PRIMARY KEY,
    doc_id text NOT NULL,
    content text NOT NULL,
    metadata jsonb,
    embedding vector(1536)
);
```

相似度查询概念：

```sql
SELECT id, content
FROM document_chunks
ORDER BY embedding <=> :query_embedding
LIMIT 5;
```

说明：

```text
embedding 存文档向量。
query_embedding 是用户问题向量。
ORDER BY 距离排序。
LIMIT 5 返回 top-5。
```

## 7. 检索评估

检索要评估：

1. 正确文档是否被召回。
2. 正确片段排名是否靠前。
3. 返回内容是否足够回答问题。
4. 是否返回大量噪声。
5. 是否遵守权限过滤。

常见指标：

- Recall@k
- Precision@k
- MRR
- 人工标注命中率
- 最终回答正确率

【核心】RAG 质量首先取决于检索质量。

## 8. 常见误区

1. 以为向量相似就一定答案正确。
2. chunk 随便按字数切。
3. 不存 metadata。
4. 不做权限过滤。
5. top-k 固定不调。
6. 忽略关键词检索价值。
7. 不评估检索，只评估最终回答。
8. 文档更新后不重新入库。
